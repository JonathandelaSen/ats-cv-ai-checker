# AI-assisted work journal

## Goal

Create a work journal feature where users can capture what they do at work or in projects with very low friction, optionally assisted by AI, and turn those entries into highlights and CV-ready bullet candidates.

The journal should create short-term value through highlights and bullets, while building a personal work-memory database that can later power AI questions, broader app context, CV edit suggestions, and long-term career planning.

## Principles

- Capturing information should be fast and should not feel like administrative work.
- AI helps with writing, synthesis, and better follow-up questions, but the user keeps editorial control.
- The journal is independent from CVs. CVs are only used to suggest initial contexts.
- Work information can be sensitive: the MVP is strictly private per user.
- Entries should not be forced to sound like measurable achievements. The highlight layer groups evidence and creates professional value later.

## Core Concepts

### Contexts

A context represents the place or area an entry belongs to.

MVP types:

- `employment`: job/employment.
- `project`: project.

Contexts are first-class journal entities, independent from CVs. On the first visit to the Journal, the app may read template CVs to detect experiences and projects, but it must not store permanent links to `cv_id`, experience indexes, or specific CV items.

Recommended initial behavior:

- If there is a clear current experience in any template CV, automatically create that context and select it by default.
- If there is no current experience, use the most recent detected experience.
- Other detected jobs/projects appear as suggestions.
- A suggestion becomes a real context when the user uses it.
- The user can create manual jobs or projects.

MVP context management:

- Create a new job/project.
- Rename.
- Mark as current/default.
- Hide suggestions.
- Archive context.

The MVP does not need an advanced management screen; minimal management from the selector is enough.

### Entries

An entry is the journal's capture unit.

Conceptual fields:

- `context_id`: required main context.
- `date_start`: defaults to today.
- `date_end`: optional; enables a free date range.
- `topic`: optional topic inside the context, with autocomplete or suggestions.
- `input_mode`: `manual` or `ai_assisted`.
- `raw_notes`: original text written by the user.
- `final_text`: final saved text.
- `created_at`, `updated_at`.

Entry modes:

- Write as-is: the user's text is saved without AI rewriting. `raw_notes` and `final_text` may be the same.
- Help me write it: the user enters lightweight or schematic notes, AI generates a first-person factual draft, the user reviews an editable preview, and only then saves.

AI should follow the language of the user's notes by default.

### Highlights

A highlight is a persistent, editable synthesis generated from entries in a single context, or created manually by the user.

The goal of a highlight is to turn raw journal material into a professional artifact that can be reused for CVs, interviews, or career reflection.

Conceptual fields:

- `context_id`: one context only.
- `title`.
- `summary`.
- `date_start`, `date_end`.
- `status`: `proposed`, `saved`, `discarded`.
- `source_entry_ids`: source entries used to generate it. Always stored when the highlight comes from entries.
- `candidate_bullets`: multiple bullet candidates.
- `detected_topics` or optional tags.
- `follow_up_questions`: questions suggested by AI.
- `additional_evidence`: user answers to follow-ups and extra notes.
- `created_at`, `updated_at`.

Statuses:

- `proposed`: generated by AI and pending review.
- `saved`: accepted/edited by the user.
- `discarded`: discarded; hidden by default.

Rules:

- A highlight belongs to a single context.
- A highlight can be created manually from scratch.
- If AI detects a similar existing highlight, it should propose merging or adding evidence instead of automatically creating a duplicate.
- Follow-up answers are stored as additional evidence. The user can then request an update/regeneration of the highlight and review the result before saving.

## MVP Flows

### First Journal Visit

1. The app detects possible contexts from all template CVs.
2. It automatically creates/selects the most likely current context.
3. It shows the capture form ready to write.
4. It shows discreet suggestions for other detected contexts.

### Create a Manual Entry

1. User confirms or changes context.
2. Date defaults to today. The user can change it to a range.
3. User optionally enters a topic.
4. User selects "Write as-is".
5. User writes the text.
6. User saves.

### Create an AI-assisted Entry

1. User confirms context, date/range, and optional topic.
2. User selects "Help me write it".
3. User writes schematic notes.
4. User clicks "Draft".
5. AI generates a first-person factual draft.
6. User edits the preview.
7. User saves the entry.

### Generate Highlights

1. User clicks "Generate highlights".
2. User chooses context and date range.
3. AI analyzes entries for that context/range.
4. AI proposes new highlights or merges with existing highlights.
5. Each highlight shows:
   - title;
   - summary;
   - period;
   - source entries;
   - bullet candidates;
   - follow-up questions.
6. User saves, edits, discards, or answers questions.

### Update a Highlight With Follow-ups

1. User answers one or more questions.
2. Answers are saved as additional evidence.
3. User clicks "Update highlight".
4. AI generates an improved version.
5. User reviews and saves.

## MVP UI

The feature lives as a primary sidebar item called "Journal".

Main view:

- New-entry capture at the top.
- Below it, tabs or sections for:
  - Entries.
  - Highlights.

MVP filters:

- Context.
- Date range.
- Simple text search.
- Topic.

Soft suggestions:

- Show non-invasive prompts when there are enough new entries in a context, for example: "You have 8 new entries in Edpuzzle. You can generate highlights."
- Highlight generation is manual in the MVP.

## Out of Scope for the MVP

- Chat or conversational questions over the journal.
- Publishing or sharing journal entries/highlights.
- Persistent links from contexts to specific CVs.
- Automatically applying highlights to CVs.
- Career planning.
- Cross-context highlights.
- Additional context types such as learning, open source, or freelance. These can be added later if usage requires them.

## Technical Considerations

- Information must be isolated by `user_id` and protected with RLS.
- Do not apply migrations to production from the agent; prepare and verify them locally.
- AI prompts and AI controllers must live in separate files.
- If prompts are created for entries or highlights, document them in `docs/prompts/<prompt-type>/prompt.md`.
- Always store both `raw_notes` and `final_text` to preserve raw evidence and curated text.
- Store `source_entry_ids` in highlights for traceability.
- Design highlight generation to receive entries filtered by context/range and existing highlights from the same context, so it can propose merging or adding evidence.

## Open Questions Before Implementation

- Final model/table names.
- Exact deduplication strategy for contexts suggested from CVs.
- Configurable language for CV bullets, if included in the MVP.
- Entry length limits and pagination.
- Deletion policy: hard delete or soft delete for entries/highlights.
